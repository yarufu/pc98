# 第29話：正式採用：通常立ち絵部分更新の裏画面VRAM方式

日付：2026/7/15

## 採用結果

通常`#left`と`#right`の部分更新を、ページ1で背景＋SPRの完成矩形を作ってから、
既存8KBバッファ経由でページ0へ転送する方式へ変更した。比較した1・4・8ラインのうち、
T98-NEXT、PC-9821V13、PC-9821Ra43で見た目と体感速度に差がなかったため、
処理単位数が最も少ない8ラインを正式採用した。

連続転送による即時消去であり、正式インターレース命令の`none`は従来どおり34段階で消去する。

比較試験専用命令、可変ライン数、試験pendingは削除した。左右同時変更は左、右の順に個別更新し、
背景変更やロード復帰などの全画面再描画は従来の背景全体＋左右SPR通常合成を維持する。
GameState、セーブ形式、Makefile、リンク設定、ライブラリ構成、新規大容量バッファは変更しない。

# 第27話：正式採用：裏画面VRAM経由の背景インターレース

日付：2026/7/13


## 採用結果

裏画面VRAMを使う試験方式を、正式な `#bginterlace` として採用した。
スクリプト作者がページ制御を意識する必要はなく、命令と引数は従来どおりである。
背景ファイル名だけを `GameState.bg_file` に保存し、セーブ形式はversion 4のまま変更しない。

比較試験用の `#bginterlacepage`、専用pending状態、コールバック、試験スクリプトは削除した。
演出中に `fseek()` / `fread()` を繰り返していた旧方式もソースから削除し、
試験関数 `graph98_load_g98_interlace_page()` を正式な
`graph98_load_g98_interlace()` へ改名した。

動作確認結果は次のとおり。

| 環境 | 結果 |
|---|---|
| T98-NEXT 4MHz設定 | 正常。旧方式より滑らか |
| PC-9821V13 | 正常。旧方式より滑らか |
| PC-9821Ra43 | 正常 |

画面の乱れ、ページ切り替え不具合、BGMテンポ、演出後の表示に問題はなかった。
G98全体の事前ロード時間も、旧方式と比べて体感上の明確な悪化はなかった。


## 調査したページ制御

標準グラフィックスモードでは、次のI/Oポートで2ページのGVRAMを選ぶ。

| I/Oポート | 用途 | 今回の使い方 |
|---|---|---|
| A4h | GDCが表示する画面の選択 | 常に00h（ページ0） |
| A6h | CPUとGDCがアクセスする描画画面の選択 | 裏画面は01h、表画面は00h |

実際に使用するページ制御I/OポートはA4hとA6hである。
表示ページは切り替えないため、ページフリップではない。

A4h/A6hの読み戻しはPC-9821など一部機種だけで保証されるため使用しない。
資料にページ切り替え後のフレーム待ちやI/O待ちの指定はなく、
切り替え直後の読み戻し、ダミーI/O、1フレーム待ちは追加していない。
I/O書き込みとVRAMアクセスのコンパイラによる並べ替えだけは、
`graph98_out8()` のmemory clobberで禁止した。

現行初期化は `INT 18h/AH=42h, CH=C0h` により標準640x400カラーを選び、
6Ahへ01hを書いて16色を選んでいる。このモードではA6hが有効である。
初期化直後とアプリケーション終了時にもA6h=00h、A4h=00hを明示するようにした。

参照資料：

- NEC「PC-9800シリーズ テクニカルデータブック HARDWARE編」
- 「UNDOCUMENTED 9801/9821 Vol.2」io_disp.txt
- NEC「PC-9800シリーズ テクニカルデータブック BIOS編」
- ReactOS FreeLoaderのPC-98画面初期化（モード設定後にA4h/A6hへ0を書いている）


## 実装した転送順序

```text
A4h = 00h
A6h = 01h
通常G98ローダーでページ1へ完成背景をロード
A6h = 00h

50段階繰り返す
    A6h = 01h
    ページ1の対象8ラインを8KB共通画像バッファへ読む
    A6h = 00h
    VSYNC待ち
    共通画像バッファからページ0の同じ8ラインへ書く

cleanupでA6h = 00h、A4h = 00h
```

CPUからページ0とページ1を同時参照できないため、
VRAM間を直接コピーせず `graph98_image_work[8192]` を経由する。
GRCGとEGCは使わず、B/R/G/Iの各プレーンをCPUで80バイトずつ読み書きする。
割り込み禁止は行わないため、PMD割り込みは通常どおり動作できる。

1段階は上側の偶数ライン4本と下側の奇数ライン4本の合計8本である。
上側は0,2,4,...,398、下側は399,397,395,...,1の順に進む。

| 項目 | 1段階の量 |
|---|---:|
| ページ1からRAM | 2,560B |
| RAMからページ0 | 2,560B |
| 合計VRAMアクセス | 5,120B |
| A6h切り替え | 2回 |
| VSYNC待ち | 1回 |

バッファはプレーン別にBlue 0～639、Red 640～1279、Green 1280～1919、
Intensity 1920～2559へ配置する。最大添字は2,559で、8KB内に収まる。
50段階で0～399ラインを重複・欠落なく1回ずつ転送する。


## ファイルI/Oとエラー処理

事前ロードでは通常G98ローダーを再利用する。
G98を1回開き、ヘッダとB/R/G/I全128,000Bを読み、1回閉じる。
現行の8KBチャンクでは `fread()` はヘッダ1回と画像16回、
`fopen()` 1回、`fclose()` 1回、`fseek()` 0回である。

事前ロード完了後の50段階では `fopen()`、`fseek()`、`fread()`、`fclose()` は
すべて0回である。

低層関数は `ok` と単一の `cleanup` 経路を使う。
ファイルオープン失敗、ヘッダ不正、読み込み失敗、ページ1への描画失敗は、
通常G98ローダーの失敗を受けてcleanupへ進み、A6h/A4hを0へ戻してから失敗を返す。
現在のCPU転送には失敗を返す操作がないが、将来の転送中断もcleanupへ集約する。
呼び出し側は既存背景処理と同じ `BG LOAD NG` の代替画面をページ0へ描く。

ページ状態は復帰するが、転送途中までページ0へ書いた画面内容はロールバックしない。
今回、キー入力による中断や旧背景の再読み込みは追加していない。


## 対応機種と今後の確認

必要条件は標準640×400・16色で、B/R/G/Iそれぞれ2ページのGVRAMを
利用できる環境である。PC-9801初代/U、Iプレーンを利用できない構成、
PC-9821の256色拡張モード、480/800ライン、ハイレゾモードは対象外とする。
自動判定と旧方式へのフォールバックは追加しない。

コード整理後はT98-NEXT 4MHz、V13、Ra43で、滑らかさ、ティアリング、
BGM、演出後の立ち絵・メッセージ・メニュー、タイトル復帰、通常 `#bg`、
連続演出、DOS終了画面を再確認する。
将来のワイプやブラインドには同じページ1→RAM→ページ0転送を流用できる。
中断時に表画面内容をロールバックするか、完成背景へ進めるかは未決定である。
