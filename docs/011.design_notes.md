# 第25話：可変長 #choice 実装

日付：2026/6/8

## ia16-gcc 最適化回避

可変長 #choice 実装時、

- input_wait_choice_jis()
- handle_choice_block()

で ia16-elf-gcc の `-O2` による Internal Compiler Error (ICE) が発生した。

そのため現状は以下の関数のみ局所的に

```
__attribute__((optimize("O0")))
```

# 第26話：#call / #return を追加した理由

当初は #jump のみでシナリオを作成していた。

しかし、実際に「見る」「話す」「移動」を使った
コマンド選択式ADVのサンプルを書いたところ、

- 同じ説明文
- 同じ移動失敗メッセージ
- 同じ選択肢

を何度も記述する必要があった。

例：

- 「まだ帰る時間ではない」
- 「容疑者たちが今ここに集められている」

など。

スクリプト量が増えると保守が困難になるため、
共通処理をまとめる仕組みとして
#call / #return を追加した。

## 実装方針

- #call ラベル名
  - 現在位置をコールスタックへ保存
  - 指定ラベルへ移動

- #return
  - コールスタックから復帰

## 備考

- ネスト対応
- セーブ／ロード時もコールスタック保存
- CALL_STACK_MAX = 8

# 第28話：FM効果音仕様

ADV98.EXE では、PMD の FM効果音機能を使用します。

FM効果音データは、実行ファイルと同じフォルダに配置した `SE.EFC` から読み込みます。

```
SE.EFC
```

PMD の仕様上、FM効果音番号は 0～127 を使用できます。

### システム効果音

ADV98.EXE では、効果音番号 1 をシステム効果音として予約しています。

1 : システム決定音

### SE.EFC が存在しない場合

SE.EFC が存在しない場合でも、ADV98.EXE はそのまま起動・動作を継続します。

この場合、

- FM効果音は再生されない
- BGM再生は通常どおり動作する
- タイトル画面は正常動作する
- セーブ／ロードは正常動作する
- スクリプト進行は正常動作する

という仕様です。

### 設計メモ：SE.EFC運用方針

ADV98.EXE では、FM効果音データを個別ファイルではなく、PMD標準形式の `SE.EFC` にまとめて管理します。

理由：

- PMD標準の運用方法に合わせるため
- 効果音管理を1ファイルに集約できるため
- 効果音番号で簡単に再生できるため
- `#se` 命令との相性が良いため

また、`SE.EFC` は必須ファイルではありません。

BGM再生とFM効果音は独立しているため、

```txt
PMD常駐
SE.EFCなし
```

現在は以下のタイミングで再生されます。

- タイトル画面の「はじめから」決定時
- タイトル画面の「ロード」決定時
- セーブ成功時

# 第31話：背景インターレースの裏画面VRAM方式

旧方式の `#bginterlace` は、演出中にG98内の対象範囲へ何度も
`fseek()`し、帯データを`fread()`して表画面へ書いていた。
DOSのファイルI/O時間が各段階へ混ざるため、特に低クロック環境では
段階間隔の引っ掛かりが見えやすかった。

正式方式では、演出開始前にG98全体をCPUアクセスページ1へ連続ロードする。
表示ページはページ0のまま切り替えない。演出中は、ページ1の対象8ラインを
8KB共通画像バッファへ読み、CPUアクセスページを0へ戻し、VSYNCを待ってから
同じ8ラインをページ0へ書く。GRCGとEGCは使わず、B/R/G/Iの4プレーンを
CPUで転送する。

この方式では演出中のファイルI/Oが0回になり、50段階すべての処理量が一定になる。
T98-NEXTの4MHz設定、PC-9821V13、PC-9821Ra43で正常動作し、
T98-NEXTとV13では旧方式より滑らかになった。事前ロード時間の体感上の
明確な悪化、画面の乱れ、BGMや演出後の表示への影響も確認されなかった。

ページ1からページ0へ直接コピーすることはできないため、RAMバッファを経由する。
同じ構成は、将来の背景ワイプやブラインドにも流用できる。
一方で、標準640×400・16色でB/R/G/I各2ページを利用できる環境が必要である。
対応機種の自動判定と、ページ1を持たない環境向けのフォールバックは設けない。

# 第30話：立ち絵インターレース表示

`#leftinterlace`と`#rightinterlace`を追加した。

PC-98時代のADVゲームで見られる、立ち絵を上下方向から走査線状に表示する演出を目的としている。

## 採用した方式

立ち絵の上側と下側から、別々の走査線群を移動させる方式を採用した。

- 上側は偶数ラインを上から下へ描画
- 下側は奇数ラインを下から上へ描画
- 走査線の周期は2ライン
- 走査線グループの高さは32ライン
- 1段階につき8ライン移動
- 各描画段階の終了後にVSYNCを待つ

現在の設定は次のとおり。

```c
#define GRAPH98_SPRITE_INTERLACE_PERIOD 2u
#define GRAPH98_SPRITE_INTERLACE_BOTTOM_OFFSET 1u
#define GRAPH98_SPRITE_INTERLACE_GROUP_LINES 32u
#define GRAPH98_SPRITE_INTERLACE_SWEEP_LINES 8u
```

上側と下側で描画するラインの位相を1ラインずらすことで、
演出終了時には偶数ラインと奇数ラインの両方が完成する。

演出中に全ラインが一度は描画されるため、
終了時に立ち絵全体をもう一度描画する処理は行わない。:contentReference[oaicite:0]{index=0}

## VSYNC待ち

各描画段階の終了後に`graph98_wait_vsync()`を呼び出す。

一時的にVSYNC待ちを外し、機種性能によって演出速度が変わる方式も試した。

VSYNC待ちなしでは、低速機では演出を目で追える一方、
高速な環境では演出がほぼ一瞬で完了した。

昔のADVゲームに近い機種依存の挙動にはなるが、
現在のADV98では演出を確実に見せることを優先し、
VSYNC待ちありを採用した。

また、移動量についても4ライン、8ライン、16ラインを比較した。

- 4ラインは滑らかだが、描画段階とファイル操作回数が増える
- 16ラインは速いが、演出が粗くなる
- 8ラインは速度と見た目のバランスがよい

このため、立ち絵は8ライン移動を正式設定とした。

## 不採用にした方式

当初は、立ち絵の表示済み範囲を上下から徐々に広げる方式を試した。

しかし、参考にした商用PC-98 ADVの走査線が移動する見た目とは異なったため採用しなかった。

次の方式も試したが、正式採用しなかった。

- 走査線間隔を広くした方式
- 大きな帯を高速に移動させる方式
- 演出終了後に立ち絵全体を再描画する方式
- VSYNC待ちを行わない方式
- 8ドット境界専用のSPR高速描画経路

8ドット境界専用の高速描画はコード容量が増えたが、
T98-NEXTおよび実機で体感差が確認できなかったため元に戻した。

## ファイル読み込み

初期実装では、走査線を描画するたびに1ライン単位で
`fseek()`と`fread()`を繰り返していたため、ファイルI/Oが多かった。

現在は、最大32ライン分をまとめて共通画像バッファへ読み込む。

```c
static uint8_t graph98_image_work[8192];
```

2回目以降の描画段階では、
前回と重複する32ライン全体を読み直さず、
新しく移動した8ライン分だけを読み込む。

SPR幅が最大256ドットの場合でも、

```text
256バイト × 32ライン = 8192バイト
```

となり、8KB共通画像バッファ内に収まる。

このバッファは通常背景、背景矩形復元、通常SPR、
背景インターレース、立ち絵インターレースで共用している。

画像処理は同時には実行されないため、
個別に大きなstaticバッファを持たず、
非再入の共通作業領域として使用する。:contentReference[oaicite:1]{index=1}

## メモリ方針

立ち絵インターレース専用の`malloc()`や大きなヒープ確保は行わない。

以前は画像処理ごとに個別のstaticバッファを持っていたが、
small modelのDGROUPを圧迫したため、8KBの共通画像バッファへ統合した。

この判断により、立ち絵インターレースを含む画像処理を維持しながら、
small modelを継続できている。

メモリ使用量やファイルI/O削減の詳細は、
`030.performance_notes.md`へ記録する。

## 現在の仕様

```text
命令：
#leftinterlace <SPRファイル名>
#rightinterlace <SPRファイル名>

SPR最大幅：
256ドット

インターレース対象最大高さ：
300ライン

走査線周期：
2ライン

走査線グループ：
32ライン

移動量：
8ライン

速度制御：
各描画段階でVSYNC待ち

終了時の全体再描画：
なし
```

通常の`#left`と`#right`とは独立した演出命令として扱う。

同じSPRファイル名がすでに設定されている場合でも、
インターレース命令を指定したときは演出を実行する。

# 第31話：立ち絵SPRファイル名の直接指定

日付：2026/7/12

立ち絵IDと表情名から固定ファイル名を作る方式を廃止し、スクリプトから
8.3形式のSPRファイル名を直接指定する方式へ変更した。

これにより、制服・私服・イベント専用画像を新しいenumや変換表なしで指定できる。
`none`は空文字列として保持し、立ち絵を非表示にする。セーブ形式はversion 3とし、
左右のSPRファイル名をそのまま保存する。version 2との変換処理は持たない。

# 第32話：背景G98ファイル名の直接指定

日付：2026/7/12

背景名へエンジン側で`.g98`を付加する方式を廃止し、スクリプトから
8.3形式のG98ファイル名を拡張子込みで直接指定する方式へ変更した。

GameStateには最大12文字の背景ファイル名を保存し、通常表示、インターレース表示、
立ち絵消去時の背景矩形復元へ同じ文字列をそのまま渡す。
セーブ形式はversion 4とし、version 3との変換処理は持たない。
