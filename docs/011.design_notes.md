# 第25話：可変長 #choice 実装

日付：2026/6/8

## ia16-gcc 最適化回避

可変長 #choice 実装時、

- input_wait_choice_jis()
- handle_choice_block()

で ia16-elf-gcc の `-O2` による Internal Compiler Error (ICE) が発生した。

そのため現状は以下の関数のみ局所的に

```
__attribute__((optimize("O0")))
```

# 第26話：#call / #return を追加した理由

当初は #jump のみでシナリオを作成していた。

しかし、実際に「見る」「話す」「移動」を使った
コマンド選択式ADVのサンプルを書いたところ、

- 同じ説明文
- 同じ移動失敗メッセージ
- 同じ選択肢

を何度も記述する必要があった。

例：

- 「まだ帰る時間ではない」
- 「容疑者たちが今ここに集められている」

など。

スクリプト量が増えると保守が困難になるため、
共通処理をまとめる仕組みとして
#call / #return を追加した。

## 実装方針

- #call ラベル名
  - 現在位置をコールスタックへ保存
  - 指定ラベルへ移動

- #return
  - コールスタックから復帰

## 備考

- ネスト対応
- セーブ／ロード時もコールスタック保存
- CALL_STACK_MAX = 8

# 第28話：FM効果音仕様

ADV98.EXE では、PMD の FM効果音機能を使用します。

FM効果音データは、実行ファイルと同じフォルダに配置した `SE.EFC` から読み込みます。

```
SE.EFC
```

PMD の仕様上、FM効果音番号は 0～127 を使用できます。

### システム効果音

ADV98.EXE では、効果音番号 1 をシステム効果音として予約しています。

1 : システム決定音

### SE.EFC が存在しない場合

SE.EFC が存在しない場合でも、ADV98.EXE はそのまま起動・動作を継続します。

この場合、

- FM効果音は再生されない
- BGM再生は通常どおり動作する
- タイトル画面は正常動作する
- セーブ／ロードは正常動作する
- スクリプト進行は正常動作する

という仕様です。

### 設計メモ：SE.EFC運用方針

ADV98.EXE では、FM効果音データを個別ファイルではなく、PMD標準形式の `SE.EFC` にまとめて管理します。

理由：

- PMD標準の運用方法に合わせるため
- 効果音管理を1ファイルに集約できるため
- 効果音番号で簡単に再生できるため
- `#se` 命令との相性が良いため

また、`SE.EFC` は必須ファイルではありません。

BGM再生とFM効果音は独立しているため、

```txt
PMD常駐
SE.EFCなし
```

現在は以下のタイミングで再生されます。

- タイトル画面の「はじめから」決定時
- タイトル画面の「ロード」決定時
- セーブ成功時

# 第31話：背景インターレースの裏画面VRAM方式

旧方式の `#bginterlace` は、演出中にG98内の対象範囲へ何度も
`fseek()`し、帯データを`fread()`して表画面へ書いていた。
DOSのファイルI/O時間が各段階へ混ざるため、特に低クロック環境では
段階間隔の引っ掛かりが見えやすかった。

正式方式では、演出開始前にG98全体をCPUアクセスページ1へ連続ロードする。
表示ページはページ0のまま切り替えない。演出中は、ページ1の対象8ラインを
8KB共通画像バッファへ読み、CPUアクセスページを0へ戻し、VSYNCを待ってから
同じ8ラインをページ0へ書く。GRCGとEGCは使わず、B/R/G/Iの4プレーンを
CPUで転送する。

この方式では演出中のファイルI/Oが0回になり、50段階すべての処理量が一定になる。
T98-NEXTの4MHz設定、PC-9821V13、PC-9821Ra43で正常動作し、
T98-NEXTとV13では旧方式より滑らかになった。事前ロード時間の体感上の
明確な悪化、画面の乱れ、BGMや演出後の表示への影響も確認されなかった。

ページ1からページ0へ直接コピーすることはできないため、RAMバッファを経由する。
同じ構成は、将来の背景ワイプやブラインドにも流用できる。
一方で、標準640×400・16色でB/R/G/I各2ページを利用できる環境が必要である。
対応機種の自動判定と、ページ1を持たない環境向けのフォールバックは設けない。

# 第30話：立ち絵インターレース表示

`#leftinterlace`と`#rightinterlace`は、立ち絵を上下方向から走査線状に
表示または消去する正式命令である。

## 採用した方式

A4hで表示ページをページ0に固定し、A6hでCPUアクセスページだけを切り替える
裏画面VRAM方式を採用した。演出開始前にページ1へ固定立ち絵矩形の背景をG98から
読み込み、SPR指定時は通常の透明合成を行って「背景＋立ち絵」の完成矩形を作る。
`none`ではSPRを合成せず、背景だけを完成矩形とする。

演出の各段階は次の順序で処理する。

1. A6hを1にし、対象ラインをページ1から8KB共通画像バッファへ読む
2. A6hを0に戻し、VSYNCを待つ
3. 共通画像バッファからページ0の同じ位置へ書く

ファイルI/Oと透明色判定はページ1の準備中に完了するため、最初のVSYNC待ち以降は
ページ1→RAM→ページ0の固定転送だけになる。ページフリップは行わず、表示ページは
準備中と演出中のどちらもページ0のままである。

対象矩形は左がX=64～319、右がX=320～575、共通してY=9～298の
256×290ドットである。左右の矩形は重ならないため、両方を変更する場合は左、右の順に
同じ処理を個別実行する。

走査線の表示順は次の設定で固定した。

- 上側は偶数ラインを上から下へ描画
- 下側は奇数ラインを下から上へ描画
- 走査線の周期は2ライン
- 最初の段階は32ライン
- 通常段階は8ライン、最後の段階は2ライン
- 全34段階で0～289の全ラインを1回ずつ転送

現在の設定は次のとおり。

```c
#define GRAPH98_STAND_WIDTH 256u
#define GRAPH98_STAND_HEIGHT 290u
#define GRAPH98_STAND_INTERLACE_PERIOD 2u
#define GRAPH98_STAND_INTERLACE_BOTTOM_OFFSET 1u
#define GRAPH98_STAND_INTERLACE_GROUP_LINES 32u
#define GRAPH98_STAND_INTERLACE_SWEEP_LINES 8u
```

最初は`0,2,...,30`に続いて`289,287,...,259`を転送し、最後は`288,1`を
転送する。欠落、重複、偶奇の位相エラーはいずれも0であり、終了時の全体再描画は不要である。

## VSYNC待ち

ページ1から対象段階をRAMへ読み終え、A6hを0へ戻した後に
`graph98_wait_vsync()`を呼び出す。VSYNCは全34段階で1回ずつ待つ。
表示タイミングへファイルI/O時間が混ざらないため、低速環境でも段階間隔が安定する。

## 不採用にした方式

旧方式は演出中にSPRファイルを開いたまま、段階ごとに`fseek()`と`fread()`を行い、
読み込んだ画素を透明色判定しながら表画面へ合成していた。ファイルI/Oと画素判定の時間が
VSYNC間へ混ざり、低速環境で段階間隔やBGMへ影響する可能性があったため廃止した。

正式方式ではSPR全体の通常透明合成を演出前のページ1で1回だけ行う。最初のVSYNC待ち
以降の`fopen()`、`fclose()`、`fseek()`、`fread()`、SPRアクセス、透明色判定はすべて0回である。

## バッファと転送量

新しい大容量バッファは追加せず、既存の共通画像バッファを利用する。

```c
static uint8_t graph98_image_work[8192];
```

立ち絵は1プレーン1ライン32Bであり、最初の32ラインを4プレーン保持しても、

```text
32B × 32ライン × 4プレーン = 4,096B
```

で収まる。プレーンごとの配置は1,024Bずつで、使用範囲は0～4,095である。
全演出のページ1→RAMとRAM→ページ0はそれぞれ37,120B、1段階の片方向最大は
4,096Bである。これらは`_Static_assert`で固定値とバッファ上限を保護する。

## エラー処理と再描画

背景G98またはSPRの準備に失敗した場合は演出を開始せず、表画面を変更しない。
すべての正常・失敗経路は`graph98_restore_default_pages()`を通り、A6h=0、A4h=0へ戻る。
旧方式への自動フォールバックは行わない。

GameStateにはSPRファイル名だけを保存し、pending状態は保存しない。ロード復帰とメニュー後は
完成画面を通常描画する。Hキーでは通常はメッセージ枠矩形だけを背景から復元し、立ち絵との
重なりまたは復元失敗時だけ完成画面を通常描画する。演出は再実行せず、セーブ形式とバージョンは
変更しない。

## 現在の仕様

```text
命令：
#leftinterlace <SPRファイル名|none>
#rightinterlace <SPRファイル名|none>

対象矩形：
左 X=64～319、右 X=320～575、Y=9～298

完成矩形：
SPR指定は背景＋立ち絵、noneは背景のみ

表示ページ：
常にページ0

処理段階：
34段階（最初32ライン、通常8ライン、最後2ライン）

演出中のファイルI/Oと透明色判定：
0回

引数なし：
状態と画面を変更せず無視し、ページ0／0へ復帰

終了時の全体再描画：
なし
```

同じSPRファイル名が設定済みでも命令を指定したときは演出を実行する。
通常の`#left none`と`#right none`は8ライン単位の連続転送で即時消去し、
インターレース命令の`none`だけが上下から進む34段階の演出消去になる。

## 通常立ち絵部分更新の裏画面VRAM化

従来の通常部分更新は、ページ0へ背景矩形のB/R/G/Iプレーンを順に復元した後、
同じページ0へSPRを透明合成していた。このため`#left none`や`#right none`で、
プレーンが完成する前の状態が赤い影のように一瞬見える場合があった。

対策として、ページ1へ背景矩形を読み込み、SPR指定時は通常透明合成を済ませてから、
完成した256×290矩形を既存8KBバッファ経由でページ0へ転送する方式を比較した。
1ライン、4ライン、8ライン単位をT98-NEXT、PC-9821V13、PC-9821Ra43で確認した結果、
色化けは解消し、3方式の見た目に差はなかった。1ラインと8ラインの体感速度にも差がなく、
V13以降の実機でも差を確認できなかったため、処理単位数が最も少ない8ラインを正式採用した。

ページ1からRAMへは64＋64＋64＋64＋34ラインの5チャンクで読み出す。ページ0では各チャンクを
8ラインずつB/R/G/Iの順に完成させ、最後の34ラインは8＋8＋8＋8＋2ラインとして処理する。
VSYNCは最初のページ0書き込み直前の1回だけである。

左右の固定矩形は重ならないため、通常左右が同時に変更された場合も左、右の順に個別転送する。
一方、背景変更やロード復帰などの全画面再描画は、背景全体へ左右SPRを通常透明合成する従来経路を
維持する。全画面背景の直後に部分更新用の背景矩形を再度読み込む冗長な処理は行わない。

通常命令の`none`は背景完成矩形を連続転送する即時消去であり、正式インターレース命令の`none`は
上下から進む段階消去である。通常8ライン転送とインターレースの通常段階8ラインは、同じライン数でも
意味と転送順が異なる。

# 第31話：立ち絵SPRファイル名の直接指定

日付：2026/7/12

立ち絵IDと表情名から固定ファイル名を作る方式を廃止し、スクリプトから
8.3形式のSPRファイル名を直接指定する方式へ変更した。

これにより、制服・私服・イベント専用画像を新しいenumや変換表なしで指定できる。
`none`は空文字列として保持し、立ち絵を非表示にする。セーブ形式はversion 3とし、
左右のSPRファイル名をそのまま保存する。version 2との変換処理は持たない。

# 第32話：背景G98ファイル名の直接指定

日付：2026/7/12

背景名へエンジン側で`.g98`を付加する方式を廃止し、スクリプトから
8.3形式のG98ファイル名を拡張子込みで直接指定する方式へ変更した。

GameStateには最大12文字の背景ファイル名を保存し、通常表示、インターレース表示、
立ち絵消去時の背景矩形復元へ同じ文字列をそのまま渡す。
セーブ形式はversion 4とし、version 3との変換処理は持たない。
