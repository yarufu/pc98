# small model からの変更

small modelは、

- コード/データが扱いやすい(コードとデータがそれぞれ最大64KB（合計128KB）)
- near pointer中心で速い
- ia16-elf-gccでも比較的安定
- 現状すでに動いている

という利点がある。

medium/large系にすると、

- far pointer増加
- 呼び出しやポインタアクセスが重くなる可能性
- バグ原因が増える
- 実機デバッグが面倒

なので、速度目的でmodel変更は難しいかも？

今のところ、model変更は、

```
メモリが足りなくなった時の最後の手段
```

でいく予定。

## ちなみに
tiny model時代は、printf 追加だけでメモリ不足・リンク不可になりやすかった。

理由は、

- tiny model は コード＋データ＋スタックが全部64KB内
- printf はライブラリが大きい
- 文字列処理・数値変換処理も引っ張ってくる
- ADV98は画像名・状態管理・スクリプト処理ですでに容量を食う

ので、printf 1個でもかなり重いです。

small modelにしたことで、

- コード領域
- データ領域

が分かれたので、かなり余裕が出た感じ。

# 第32話：背景インターレース追加時のメモリ不足とI/O最適化

日付：2026/7/11

`#bginterlace` 実装時、背景インターレース用として  
2,560バイトの `static` バッファを新たに追加したところ、  
T98-NEXTで以下の問題が発生した。

- `#bginterlace` が極端に遅い
- 既存の立ち絵インターレースまで遅くなる
- BGMのテンポが遅くなる
- メッセージが文字化けする

調査の結果、背景インターレース処理の配列範囲外アクセスや、  
DS、ES、SS、SPの破壊は確認されなかった。

最有力原因は、small modelのDGROUP圧迫だった。

### DGROUPの状態

背景インターレース追加前後の主な数値は以下。

| 項目 | 追加前 | 追加後 |
| --- | ---: | ---: |
| `.bss` | 40,264B | 42,824B |
| DGROUP静的ペイロード | 59,576B | 62,168B |
| 最低スタック予約を除いた余裕 | 5,160B | 2,568B |

現在のプログラムでは、`script.txt` を開いたまま画像ファイルを開く。

使用しているlibcでは、通常1ファイルにつき約1,024バイトの  
stdioバッファが必要になる。

DGROUPの空きが約2,568バイトしかない状態では、  
2本目のFILEバッファ確保が失敗し、  
1バイトの内部バッファを使用する低速動作へ  
フォールバックした可能性が高い。

これにより、今回追加した背景処理だけでなく、  
既存のSPR読み込みも極端に遅くなったと考えられる。

文字化けについては、ヒープとスタックの接近による  
メモリ破壊が最有力と判断した。

### 画像バッファの共通化

変更前は、画像処理ごとに以下の `static` バッファを持っていた。

| 用途 | サイズ |
| --- | ---: |
| 通常G98 | 2,560B |
| 背景インターレース | 2,560B |
| G98矩形読み込み | 80B |
| 通常SPR | 8,192B |
| 旧SPR演出（削除済み） | 8,192B |
| SPRインターレース | 8,192B |

これらの画像処理は同時には実行されないため、  
8,192バイトの共通画像バッファ1本へ統合した。

```c
static uint8_t graph98_image_work[8192];
```

共通化後の結果は以下。

| 項目 | 共通化前 | 共通化後 | 改善 |
| --- | ---: | ---: | ---: |
| `.bss` | 42,824B | 21,240B | -21,584B |
| DGROUP静的ペイロード | 62,168B | 40,584B | -21,584B |
| 最低スタック予約を除いた余裕 | 2,568B | 24,152B | +21,584B |

共通化後、通常版・EGC版の両方で以下が解消した。

- メッセージ文字化け
- 既存立ち絵インターレースの速度低下
- BGMテンポ低下

この結果から、small modelそのものが使用不可能だったのではなく、  
重複した `static` バッファによってDGROUPを圧迫していたことが  
主原因と判断できる。

画像処理用の大きな作業領域は、非再入であることを確認したうえで、  
可能な限り共通化する。

### 背景インターレースの重複I/O削減

初期実装では、32ラインの帯を8ラインずつ移動しながら、  
毎回32ライン全体を読み直していた。

初期実装の処理量は以下。

| 項目 | 初期実装 |
| --- | ---: |
| sweep数 | 50回 |
| `fseek` | 400回 |
| `fread` | 400回 |
| 総読み込み量 | 993,280B |
| VRAM書き込み量 | 496,640B |

前回の帯と24ライン重複していたため、  
最初のsweepのみ上下32ラインを読み込み、  
2回目以降は新しく進んだ上下8ラインだけを  
読み込む方式へ変更した。

最適化後は以下。

| 項目 | 最適化後 |
| --- | ---: |
| sweep数 | 47回 |
| `fseek` | 376回 |
| `fread` | 376回 |
| 総読み込み量 | 256,000B |
| VRAM書き込み量 | 128,000B |

読み込み量とVRAM書き込み量を約74％削減できた。

機械的な被覆確認では、全400ラインについて以下を確認した。

- 欠落ラインなし
- 重複描画なし
- 偶数・奇数ラインの位相違反なし
- 最終通常背景再読み込みなし
- 完成画像は通常の `#bg` と一致

### small model継続時の注意

今回の共通化によりDGROUP側には余裕が戻ったため、  
small modelは継続可能と判断した。

一方、コードセグメント側の余裕は非常に少ない。

| 構成 | 64KBまでの残量 |
| --- | ---: |
| 通常版／DIRECT版 | 約800B |
| EGC版 | 約160B |

特にEGC版は、リンクが成功しても64KB境界を越えている可能性があるため、  
今後の変更時はリンカーマップでコード末尾を確認する。

リンク成功だけで安全と判断しない。

# 第32話：EGCによる矩形塗りつぶし

## 概要

`graph98_boxfill()` の単色矩形塗りつぶしについて、EGCを使用する実験版を追加した。

通常版では従来どおり、VRAMのB/R/G/I各プレーンへ直接書き込む。

EGC版では、条件を満たす矩形に対してEGCを使用し、CPUから代表プレーンへ書き込むことで4プレーンをまとめて更新する。

現時点でEGC対応しているのは、次の処理だけである。

- `graph98_boxfill()` による単色矩形塗りつぶし

次の処理は、まだEGC化していない。

- G98背景画像の読み込み
- SPR立ち絵描画
- 透過SPR描画
- 立ち絵インターレース表示
- 文字描画
- 水平線・垂直線・枠線

---

## ビルド方法

通常のDIRECT版を作る場合：

```sh
make clean && make
```

明示的にDIRECT版を別名で作る場合：

```sh
make clean && make USE_EGC=0 TARGET=build/ADV98D.EXE
```

EGC版を作る場合：

```sh
make clean && make USE_EGC=1 TARGET=build/ADV98E.EXE
```

通常の `make` では `USE_EGC=0` が使用される。

`USE_EGC=0` の場合：

- `src/egc98.c` をリンクしない
- EGC用コードを実行ファイルへ含めない
- 従来のVRAM直接描画だけを使用する

`USE_EGC=1` の場合：

- `src/egc98.c` をリンクする
- `graph98_boxfill()` からEGC描画を使用できる
- EGCを使用できない場合は従来のDIRECT描画へ戻る

コンパイルオプションは両方とも同じである。

```text
-O2
-mcmodel=small
-march=v30
```

---

## EGCを使用する条件

EGC版でも、すべての矩形をEGCで描画するわけではない。

次の条件をすべて満たす場合だけ、EGC経路を使用する。

- `USE_EGC=1` でビルドされている
- BIOSのEGC存在フラグがON
- クリップ後の矩形が画面内
- 幅と高さが0ではない
- 左端のX座標が16ドット境界
- 矩形の幅が16ドットの倍数
- `egc98_boxfill_aligned16()` が正常終了する

条件を満たさない場合は、従来のVRAM直接描画へフォールバックする。

矩形の幅や面積による最低サイズのしきい値は設けていない。そのため、16×16の小矩形でも、座標と幅が整列していればEGC対象になり得る。

---

## EGC存在判定

EGCの存在は、BIOSワークエリアのフラグを確認して判定する。

EGCフラグがOFFの場合は、EGC用I/Oポートへアクセスせず、従来のDIRECT描画を使用する。

このため、EGC非搭載機でもEGC版の実行自体は可能な設計としている。ただし、未確認機種を含めた完全な互換性を保証するものではない。

---

## EGC開始・終了処理

EGCモードの開始時と終了時だけ、割り込みを一時的に禁止する。

矩形を描画している間は割り込みを許可する。

EGC終了時には、EGCレジスタとアクセスモードを通常VRAM描画用の状態へ戻す。

これにより、EGC描画後も次の処理を従来どおり実行できる。

- DIRECTによる矩形描画
- SPR描画
- 文字描画
- G98背景描画
- DOSへの復帰

各走査線の描画前には、EGCシフタ位置とビット長を明示的に再設定する。

## ベンチマーク

`tests/egc_bench.c` を使用し、DIRECT描画とEGC描画を比較した。

ベンチ用EXEの作成：

```sh
make egc-bench
```

EGC単体テスト用EXEの作成：

```sh
make egc-test
```

実機のDOS時刻取得は分解能が粗いため、各矩形を多数回繰り返して計測した。

PC-9821V13では、次の傾向になった。

| 矩形サイズ | DIRECT | EGC | 傾向 |
|---:|---:|---:|---|
| 16×16 | 約12秒 | 約18秒 | 小矩形ではEGCが遅い |
| 64×64 | 約12秒 | 約5秒 | EGCが速い |
| 320×200 | 約12秒 | 約2秒 | EGCが大幅に速い |
| 640×400 | 約12秒 | 約2秒 | EGCが大幅に速い |

PC-9821Ra43でも、おおむね同じ傾向を確認した。ただし、CPUが高速なため計測値が0秒になる場合があり、正確な絶対時間の比較には向いていない。

この結果から、次の傾向がある。

- 小さな矩形ではEGCの初期設定コストが目立つ
- 大きな矩形では4プレーン分のCPU書き込みを削減できる
- 全画面や大きな矩形ではEGCの効果が大きい
- 機種によって速度の境界が異なる可能性がある
- ベンチ結果は全PC-98共通の最適境界を示すものではない

16×16の結果は多数回ループした合計値であり、実際のゲーム中に1～2回描画する程度なら、DIRECTとの差は体感上ほぼ無視できる。

このため、現時点では矩形サイズによるしきい値を設けず、実装を単純に保っている。

---

## 現在のADV98内でEGCになる主な処理

現在のADV98では、小さな整列矩形はほとんど使用していない。

EGC対象になる主な矩形は次のとおり。

- 640×400の全画面消去
- 640×300の代替背景上部
- 640×100の代替背景下部
- 64×400の左右装飾部分
- 512×56の横長矩形
- 512×110の横長矩形
- 512×34の横長矩形
- 512×100の床領域

メッセージウインドウ、選択帯、通知表示などは、X座標または幅が16ドット整列していないため、現在はDIRECT描画になる。

## 今後の候補

今後EGC化を検討できる処理：

- 長い水平線の高速化
- 背景の単色消去
- VRAM間コピー
- 事前に4プレーン形式へ変換した画像の転送
- 大きな不透明画像の描画

SPRは、ファイル読み込み、透明色判定、色変換などの処理も含むため、単純にEGCへ置き換えるだけでは高速化できない可能性がある。

小さなUI部品や瞬き用の目パーツなどは、描画回数と実際の速度を確認してからEGC化を判断する。
