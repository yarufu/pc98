# small model からの変更

small modelは、

- コード/データが扱いやすい(コードとデータがそれぞれ最大64KB（合計128KB）)
- near pointer中心で速い
- ia16-elf-gccでも比較的安定
- 現状すでに動いている

という利点がある。

medium/large系にすると、

- far pointer増加
- 呼び出しやポインタアクセスが重くなる可能性
- バグ原因が増える
- 実機デバッグが面倒

なので、速度目的でmodel変更は難しいかも？

今のところ、model変更は、

```
メモリが足りなくなった時の最後の手段
```

でいく予定。

## ちなみに
tiny model時代は、printf 追加だけでメモリ不足・リンク不可になりやすかった。

理由は、

- tiny model は コード＋データ＋スタックが全部64KB内
- printf はライブラリが大きい
- 文字列処理・数値変換処理も引っ張ってくる
- ADV98は画像名・状態管理・スクリプト処理ですでに容量を食う

ので、printf 1個でもかなり重いです。

small modelにしたことで、

- コード領域
- データ領域

が分かれたので、かなり余裕が出た感じ。

# VRAM直接描画をやめてEGCを使う？

これは かなり有力。

ADV98で重そうなのは主にここ。

- 背景の全画面描画
- 背景矩形復元
- 立ち絵SPR描画
- ワイプ・インターレース演出

今は確か、

```
CPUでピクセル/プレーンを計算して、VRAM 4プレーンへ直接書く
```

という形なので、PC-98では遅くなりやすい。

EGCを使うと、特に以下が速くなる可能性がある。

- 矩形塗りつぶし
- 横方向の連続転送
- マスク付き描画
- VRAM間コピー
- パターン転送

ただし、透明色つきSPR描画を全部EGC化 は少し難しそう。

最初に狙うなら、

背景描画・矩形復元・ワイプ系をEGC対応

が現実的？

## PC-98 の世代

| 機種          | EGC | 動作            |
| ----------- | --- | ------------- |
| PC-9801VM   | ×   | 動かない（EGC使用不可） |
| PC-9801VX   | ○   | 動作            |
| PC-9801UX   | ○   | 動作            |
| PC-9801RX   | ○   | 動作            |
| PC-9801RA   | ○   | 動作            |
| PC-9821シリーズ | ○   | 動作            |

なので、

「VX以降対応」

という表記が一番正確だと思われる。


## VM以降

VMは名機だったので、

- GRCGだけで描く
- EGCを使わない
- 640KB以内

という設計が多かったためです。

つまり

```
VMでも動く = 一番互換性が高い
```

という意味？
