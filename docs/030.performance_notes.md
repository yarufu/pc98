# 第32話：EGC対応の検討・部分実装・削除

日付：2026/07/12

## 結論

`graph98_boxfill()`の一部をEGC対応して実機検証を行ったが、現時点では通常版とEGC版を並行して保守する利点が小さいと判断した。

そのため、EGC専用コード、テスト、ベンチマーク、ビルド設定を削除し、通常版の`ADV98.EXE`一本へ戻した。

現在の正式ビルドは次のとおり。

```sh
make clean && make
```

出力：

```text
build/ADV98.EXE
```

削除前のEGC実装は、Gitタグに保存している。

```text
EGC部分的実装削除前
```

将来、矩形転送、フェード、画面効果などでEGCによる明確な効果が期待できる場合に再検討する。

---

## EGCを検討した理由

ADV98では、CPUが画像データを処理し、VRAMのB/R/G/I各プレーンへ直接書き込んでいる。

特に負荷が大きいと考えられた処理は次のとおり。

- 背景の全画面描画
- 背景の矩形復元
- 立ち絵SPR描画
- 背景・立ち絵のインターレース演出
- 大きな単色矩形の塗りつぶし

EGCを使用すると、CPUが4プレーンへ個別に書き込む処理を減らせるため、次の処理で高速化できる可能性がある。

- 大きな矩形の塗りつぶし
- 横方向の連続描画
- マスク付き描画
- VRAM間コピー
- パターン転送

一方、透明色付きSPRは、ファイル読み込み、透明色判定、色ごとのプレーン生成なども必要になるため、単純にEGCへ置き換えるだけでは高速化しにくい。

当初は背景描画、矩形復元、インターレース処理などへの展開も検討したが、実際に実装したのは`graph98_boxfill()`の単色矩形描画だけである。

---

## 対応機種についての考え方

EGC版を正式採用する場合は、対応機種をEGC搭載機へ限定する必要がある。

調査時には、主な対象を次の世代として想定していた。

```text
PC-9801VX以降
PC-9821シリーズ
```

PC-9801VMなどのEGC非搭載機や、EPSON互換機を含めた互換性を維持するには、通常のVRAM直接描画版も残す必要がある。

このため、EGC版を正式採用すると、

```text
通常版
EGC版
```

の2種類をビルド、配布、動作確認する必要が生じる。

現行コードにはGRCG実装はなく、通常版はVRAMのB/R/G/I各プレーンへ直接書き込む方式である。

---

## 部分実装した内容

EGC対応したのは、次の処理だけである。

- `graph98_boxfill()`による単色矩形塗りつぶし

次の処理はEGC化していなかった。

- G98背景画像の読み込み
- 背景の矩形復元
- SPR立ち絵描画
- 透過SPR描画
- 背景インターレース
- 立ち絵インターレース
- 文字描画
- 水平線・垂直線・枠線

通常版では従来どおり、VRAMのB/R/G/I各プレーンへ直接書き込む。

EGC版では、条件を満たす矩形に対してEGCを使用し、CPUから代表プレーンへ書き込むことで4プレーンをまとめて更新していた。

---

## EGC経路を使用していた条件

削除前のEGC版では、次の条件をすべて満たす場合だけEGC経路を使用していた。

- `USE_EGC=1`でビルドされている
- BIOSのEGC存在フラグがON
- クリップ後の矩形が画面内
- 幅と高さが0ではない
- 左端のX座標が16ドット境界
- 矩形の幅が16ドットの倍数
- `egc98_boxfill_aligned16()`が正常終了する

条件を満たさない場合は、従来のVRAM直接描画へ戻る設計だった。

矩形サイズによる最低しきい値は設けず、16×16の小矩形でも座標と幅が整列していればEGC対象としていた。

---

## EGC存在判定と初期化

EGCの存在は、BIOSワークエリアのフラグを確認して判定していた。

EGCフラグがOFFの場合はEGC用I/Oポートへアクセスせず、従来のVRAM直接描画を使用していた。

EGCモードの開始時と終了時だけ割り込みを一時的に禁止し、描画中は割り込みを許可していた。

終了時にはEGCレジスタとアクセスモードを通常VRAM描画用の状態へ戻していた。

各走査線の描画前には、EGCシフタ位置とビット長を再設定していた。

これにより、EGC描画後も次の処理を通常どおり実行できた。

- VRAM直接描画
- SPR描画
- 文字描画
- G98背景描画
- DOSへの復帰

---

## 動作確認環境

次の環境で、EGC単体テストとADV98本体の表示を確認した。

- T98-NEXT
- PC-9821V13
- PC-9821Ra43

確認内容：

- BIOSのEGCフラグ判定
- EGCによる16色の矩形描画
- 各走査線のシフタ再設定
- EGC終了後のVRAM直接描画
- タイトル画面
- メッセージ表示
- 選択肢
- システムメニュー
- セーブ／ロード
- SPR・文字表示
- DOSへの正常復帰
- ハングや画面破壊がないこと

---

## ベンチマーク結果

削除前の`tests/egc_bench.c`を使用し、VRAM直接描画とEGC描画を比較した。

PC-9821V13では、次の傾向になった。

| 矩形サイズ | VRAM直接描画 | EGC | 結果 |
|---:|---:|---:|---|
| 16×16 | 約12秒 | 約18秒 | 小矩形ではEGCが遅い |
| 64×64 | 約12秒 | 約5秒 | EGCが速い |
| 320×200 | 約12秒 | 約2秒 | EGCが大幅に速い |
| 640×400 | 約12秒 | 約2秒 | EGCが大幅に速い |

PC-9821Ra43でも、おおむね同じ傾向だった。

ただし、Ra43ではCPUが高速なため計測値が0秒になる場合があり、正確な絶対時間の比較には向いていなかった。

ベンチ結果から、次の傾向を確認した。

- 小さな矩形ではEGCの初期設定コストが目立つ
- 大きな矩形では4プレーン分のCPU書き込みを削減できる
- 全画面や大きな矩形ではEGCの効果が大きい
- 有利になる矩形サイズは機種によって異なる可能性がある
- この結果は全PC-98共通の最適境界を示すものではない

16×16の結果は多数回描画(五十万回のループ処理)した合計値である。

ゲーム中に1～2回描画する程度なら、VRAM直接描画との差は体感上ほぼ無視できる。

## ADV98内でEGC対象だった処理

削除前のADV98で、EGC対象になり得た主な矩形は次のとおり。

- 640×400の全画面消去
- 640×300の代替背景上部
- 640×100の代替背景下部
- 64×400の左右装飾部分
- 512×56の横長矩形

一方、メッセージウインドウ、選択帯、通知表示などは、X座標または幅が16ドット境界に揃っていないため、EGC版でもVRAM直接描画を使用していた。

ADV98の通常動作では、全画面消去など一部の処理以外でEGCの恩恵を受ける場面が限られていた。

## EGC対応を削除した理由

実装と実機確認の結果、次の理由から現在はEGC対応を削除した。

- EGC化できていたのは`graph98_boxfill()`だけだった
- 背景、SPR、インターレースなど主要な画像描画は高速化されなかった
- 通常版でも現在のADVゲーム用途には十分な速度だった
- メッセージウインドウなど主要UIは16ドット境界に揃っておらず、EGC対象にならなかった
- 通常版とEGC版を別々にビルド、配布、動作確認する必要があった
- EGC非搭載機や互換機への配慮が必要だった
- 今後の機能追加時に2経路を保守する負担が増える
- 部分実装を残すより、通常版一本の方がソースとMakefileが分かりやすい

EGCコード削除前後で通常版EXEを比較したところ、バイナリは完全に一致した。

```text
SHA-256:
904f476516bf009b6d25d0a2738ddf649af05bcafac72cbb703c73d7159e5954
```

結果：

- `cmp`差分なし
- EXEサイズ変更なし
- コード容量変更なし
- `.data`変更なし
- `.bss`変更なし
- DGROUP変更なし
- スタック使用量変更なし

削除前の通常版では、EGCコードが`USE_EGC=0`によって完全にコンパイル対象外だったため、削除後も通常版の実行内容は変化していない。

# 第32話：small model継続とメモリ・I/O最適化

日付：2026/07/11

## 結論

ADV98では、引き続きsmall modelを使用する。

現在の正式ビルドは次のとおり。

```sh
make clean && make
```

出力：

```text
build/ADV98.EXE
```

主なコンパイルオプション：

```text
-O2
-mcmodel=small
-march=v30
```

medium modelやlarge modelへの変更は行わず、メモリ不足が発生した場合は、まず次の方法で対応する。

- 重複した作業用バッファを共通化する
- 不要なstaticデータを削減する
- ファイル読み込み回数を減らす
- 不要なコードや機能を整理する
- リンカーマップでコードとDGROUPの使用量を確認する

メモリモデルの変更は、これらの方法でも対応できなくなった場合の最後の手段とする。

---

## small modelを継続する理由

small modelには次の利点がある。

- コードとデータをそれぞれ最大64KBまで使用できる
- near pointer中心のため、far pointerより扱いやすい
- 関数呼び出しやデータアクセスが比較的軽い
- ia16-elf-gccで現在安定してビルドできている
- T98-NEXTと実機ですでに動作している

medium modelやlarge modelへ変更すると、コードやデータの上限を広げられる可能性がある。

一方で、次の問題が増える。

- far pointerやfar callが増える
- 関数呼び出しやポインタアクセスが重くなる可能性がある
- nearとfarの取り違えによる不具合が増える
- ライブラリやコンパイラ側の対応状況も確認する必要がある
- 実機での不具合調査が難しくなる

そのため、速度改善を目的としたメモリモデル変更は行わない。

---

## tiny model時代との違い

以前使用していたtiny modelでは、コード、データ、BSS、スタックなどをすべて同じ64KB以内へ収める必要があった。

そのため、`printf()`のような比較的大きなライブラリ関数を追加しただけでも、リンクできなくなることがあった。

`printf()`を使用すると、文字列処理や数値変換などのライブラリコードも追加される。

ADV98は次の処理ですでにメモリを使用している。

- スクリプト処理
- セーブデータ
- 背景・立ち絵ファイル名
- ゲーム状態
- 文字描画
- ファイル入出力
- 画像処理用バッファ

small modelへ変更したことで、コード領域とデータ領域が分離され、tiny model時代より余裕を確保できた。

---

## DGROUP圧迫による不具合

背景画像用の新しい描画処理を追加した際、2,560バイトのstaticバッファを追加したところ、次の問題が発生した。

- 背景画像の描画が極端に遅くなる
- 既存の立ち絵描画も遅くなる
- BGMのテンポが遅くなる
- メッセージが文字化けする

配列範囲外アクセスや、DS、ES、SS、SPの破壊は確認されなかった。

調査の結果、最有力原因はsmall modelのDGROUP圧迫だった。

### 発生時のDGROUP

当時の測定値は次のとおり。

| 項目 | 追加前 | 追加後 |
|---|---:|---:|
| `.bss` | 40,264B | 42,824B |
| DGROUP静的ペイロード | 59,576B | 62,168B |
| 最低スタック予約を除いた余裕 | 5,160B | 2,568B |

ADV98では、スクリプトファイルを開いたまま画像ファイルを開く場合がある。

使用しているlibcでは、通常は1ファイルにつき約1,024バイトのstdioバッファが必要になる。

DGROUPの空きが約2,568バイトしかない状態では、2本目のファイル用バッファを十分に確保できず、小さな内部バッファを使用する低速動作へ移行した可能性が高い。

この影響は新しく追加した処理だけでなく、既存のSPR読み込みやBGM再生にも及んだと考えられる。

文字化けについては、ヒープとスタックの接近によるメモリ破壊が原因だった可能性が高い。

---

## 画像作業用バッファの共通化

変更前は、画像処理ごとに別々のstaticバッファを持っていた。

| 用途 | サイズ |
|---|---:|
| 通常G98 | 2,560B |
| 背景用追加処理 | 2,560B |
| G98矩形読み込み | 80B |
| 通常SPR | 8,192B |
| 旧SPR演出 | 8,192B |
| SPR追加描画処理 | 8,192B |

これらの画像処理は同時には実行されない。

そのため、8,192バイトの共通画像バッファ1本へ統合した。

```c
static uint8_t graph98_image_work[8192];
```

共通化直後の結果は次のとおり。

| 項目 | 共通化前 | 共通化後 | 改善 |
|---|---:|---:|---:|
| `.bss` | 42,824B | 21,240B | -21,584B |
| DGROUP静的ペイロード | 62,168B | 40,584B | -21,584B |
| 最低スタック予約を除いた余裕 | 2,568B | 24,152B | +21,584B |

共通化後、次の問題は解消した。

- メッセージ文字化け
- 立ち絵描画の速度低下
- BGMテンポ低下
- 背景描画の異常な低速化

この結果から、small modelそのものが原因ではなく、重複したstaticバッファによるDGROUP圧迫が主原因だったと判断した。

大きな作業用バッファは、処理が非再入であり、同時使用されないことを確認したうえで共通化する。

---

## 背景インターレースの裏画面VRAM化

旧方式は、32ライン帯を8ラインずつ移動し、演出中にG98へ多数の
`fseek()` / `fread()`を行っていた。最終版の旧方式でも47段階で、
`fseek()` 376回、`fread()` 376回、合計256,000Bを読み込んでいた。
ファイルI/O時間が各段階へ混ざるため、段階間隔が不均一になりやすかった。

新方式は、演出前にG98をCPUアクセスページ1へ連続読み込みして完成させる。
その後の50段階はVRAMと8KB共通画像バッファの転送だけを行い、
演出中の`fopen()`、`fseek()`、`fread()`、`fclose()`はすべて0回である。

| 項目 | 新方式 |
|---|---:|
| 処理段階数 | 50回 |
| 1段階の表示ライン | 上4本 + 下4本 = 8本 |
| ページ1 → RAM | 2,560B/段階 |
| RAM → ページ0 | 2,560B/段階 |
| 合計VRAMアクセス | 5,120B/段階 |
| 演出中のファイルI/O | 0回 |

1段階の処理量が一定になり、T98-NEXTの4MHz設定、PC-9821V13、
PC-9821Ra43で正常動作を確認した。T98-NEXTとV13では旧方式より滑らかで、
事前ロード時間にも体感上の明確な悪化はなかった。

代わりに、標準640×400・16色でB/R/G/I各2ページを使える環境が必要になる。
表示ページ0は切り替えず、ページ1からRAMへ読んだ後にCPUアクセスページを0へ戻し、
VSYNCを待ってページ0へ書く。

---

## 立ち絵インターレースの裏画面VRAM化

比較試験した裏画面VRAM方式を正式な`#leftinterlace`と`#rightinterlace`へ採用した。
演出前にページ1へ対象領域の背景を読み込み、SPR指定時は通常の透明合成で完成矩形を作る。
`none`では背景だけを完成矩形とし、同じ転送処理で表画面の旧立ち絵を段階消去する。

動作確認結果は次のとおり。

| 試験環境 | 結果 |
|---|---|
| T98-NEXT 4MHz設定 | 正常 |
| PC-9821V13 | 正常 |
| PC-9821Ra43 | 正常 |

3環境で立ち絵表示、透明部分、BGM、左右同時処理、`none`、連続実行に問題はなかった。

| 項目 | 正式方式 |
|---|---:|
| 処理段階数 | 34回 |
| 最初の段階 | 32ライン |
| 通常段階 | 8ライン |
| 最後の段階 | 2ライン |
| ページ1 → RAM | 37,120B |
| RAM → ページ0 | 37,120B |
| 1段階最大 | 4,096B／片方向 |
| VSYNC待ち | 34回 |
| 演出中のファイルI/O | 0回 |
| 最大バッファ使用量 | 4,096B |
| 最大バッファ添字 | 4,095 |
| 新規大容量バッファ | なし |

最初のVSYNC待ち以降は、`fopen()`、`fclose()`、`fseek()`、`fread()`、
SPRファイルアクセス、透明色判定を一切行わない。演出中はページ1→RAM→ページ0の
固定転送だけである。

## 通常立ち絵部分更新の裏画面VRAM化

従来の`#left none`と`#right none`では、ページ0へ背景の4プレーンを順番に復元する途中で、
赤い影のような色が一瞬見える場合があった。ページ1で背景＋SPRの完成矩形を作り、
RAM経由でページ0へ小単位転送する試験実装を比較した。

| 試験環境 | 結果 |
|---|---|
| T98-NEXT | 1・4・8ラインの見た目と体感速度に差なし。赤い影は解消 |
| PC-9821V13 | 1ラインと8ラインの見た目・体感速度に差なし |
| PC-9821Ra43 | 1ラインと8ラインの見た目・体感速度に差なし |

3環境とも1・4・8ライン方式の見た目に差はなく、V13以降の実機でも1ラインと8ラインの
速度差を確認できなかった。このため、ページ0への処理単位数が最も少ない8ライン方式を
通常`#left`と`#right`の部分更新へ正式採用した。

| 項目 | 正式8ライン方式（片側1回） |
|---|---:|
| 対象ライン | 290 |
| 小単位数 | 37 |
| RAMチャンク | 5 |
| チャンク構成 | 64＋64＋64＋64＋34ライン |
| 最終小単位 | 2ライン |
| ページ1 → RAM | 37,120B |
| RAM → ページ0 | 37,120B |
| 最大バッファ使用量 | 8,192B |
| 最大バッファ添字 | 8,191 |
| VSYNC待ち | 1回 |
| ライン欠落 | 0 |
| ライン重複 | 0 |
| 最初のVSYNC以降のファイルI/O | 0回 |
| 最初のVSYNC以降の透明色判定 | 0回 |

左右同時変更は全画面再描画へ戻さず、左、右の順に同じ部分更新を個別実行する。
背景変更、ロード復帰、メニュー後、タイトルからのロード直後など、背景全体を描画する経路では
ページ1へ背景＋左右SPRを完成合成してから表示し、背景矩形の再読み込みは行わない。Hキーでは通常は
メッセージ枠矩形だけを背景から復元し、立ち絵との重なりまたは復元失敗時だけ全画面再描画する。

## 通常画像読み込みのI/O削減

背景、背景矩形復元、SPRの通常読み込みについても、8KB共通画像バッファを利用して複数ラインをまとめて読み込む方式へ変更した。

これにより、画像データの総量を変えずに、`fseek()`や`fread()`の呼び出し回数を削減した。

DOS環境では、同じデータ量でもファイル関数の呼び出し回数が多いほど遅くなる可能性がある。

そのため、次の方針を基本とする。

- 可能な限り複数ラインをまとめて読み込む
- 1ラインごとの`fseek()`を避ける
- 読み込み総量だけでなく、関数呼び出し回数も確認する
- CPU処理とディスクI/Oを分けて考える

非ゼロのfar textは存在しない。

small modelを継続できるだけの余裕はあるが、コードとDGROUPの両方について、今後も定期的な確認が必要である。

## 今後の注意点

small modelで開発を継続する場合は、次の点に注意する。

- 大きなstatic配列を安易に追加しない
- 作業用バッファは共通化できないか確認する
- 同時に開くファイル数を意識する
- stdio内部バッファ用のヒープも残す
- ヒープとスタックの余裕を確保する
- リンク成功だけで安全と判断しない
- リンカーマップでコード末尾を確認する
- `.data`と`.bss`を合わせたDGROUP使用量を確認する
- 実機でBGM速度、文字化け、画像読み込み速度も確認する

異常な速度低下や文字化けが発生した場合は、描画アルゴリズムだけでなく、DGROUP不足によるstdioの低速化やヒープ・スタックの接近も疑う。

## 通常背景の裏画面完成合成方式

通常背景変更とロード復帰の中間色露出を避けるため、ページ1で背景＋左SPR＋右SPRを
完成させ、VSYNCでページ1を表示している間にページ0へ全画面複製し、次のVSYNCで
ページ0へ戻す方式を通常`#bg`とロード復帰へ正式採用した。背景変更とロード復帰で
見えていた赤い中間色は消え、T98-NEXTで正常表示を確認した。

立ち絵インターレース予定側はページ1へ通常合成せず、完成背景を表示した後に既存34段階処理を
実行する。左右両方なら左、右の順である。完成合成に失敗した場合は0／0へ戻し、従来の
背景＋通常SPR直接描画へフォールバックするため、既存の画像読み込み失敗表示も維持される。

全画面コピーは既存8KB画像バッファを共用し、プレーンごとに連続チャンク転送する。

| 項目 | 理論値 |
|---|---:|
| 画面 | 640×400、4プレーン |
| 1ライン | 80B／プレーン |
| 1プレーン | 32,000B |
| 4プレーン | 128,000B |
| 1プレーンのチャンク | 102＋102＋102＋94ライン |
| RAMチャンク | 16 |
| 最大バッファ使用量 | 8,160B |
| 最大バッファ添字 | 8,159 |
| ページ1 → RAM | 128,000B |
| RAM → ページ0 | 128,000B |
| 合計VRAM橋渡し量 | 256,000B |
| VSYNC待ち | 2回 |
| 最初のVSYNC以降のファイルI/O | 0回 |
| 最初のVSYNC以降の透明色判定 | 0回 |
| 最初のVSYNC以降のSPRアクセス | 0回 |

A4hの正常時書き込み列は`0, 1, 0, 0`で4回、初期状態を0とした実値遷移は2回である。
最後の0は共通cleanupによる保証である。A6hの正常時書き込み列は
`1, (1, 0)×16, 0`で34回、初期状態を0とした実値遷移は32回である。
正常終了も準備失敗もA4h=0、A6h=0を保証する。

追加の回帰確認ではT98-NEXT（4MHz相当を含む）、PC-9821V13、PC-9821Ra43で、
赤い影、プレーン途中、立ち絵途中、ページ切り替えのちらつき、黒画面、上下のずれ、
ページ1表示中の安定性、ページ0復帰時のちらつき、待ち時間、BGMへの影響を比較する。

試験版から正式版へのビルド結果は次のとおり。新しい大容量staticデータやローカル配列、
ライブラリ関数、Makefile・リンク設定は追加していない。

| 項目 | 試験版 | 正式版 | 差 |
|---|---:|---:|---:|
| 実コード末尾 | `0xE26A` | `0xE290` | +38B |
| アライン後コード末尾 | `0xE270` | `0xE290` | +32B |
| コード64KBまでの残量 | 7,568B | 7,536B | -32B |
| EXEサイズ | 77,184B | 77,200B | +16B |
| `.text` | 77,152B | 77,168B | +16B |
| `.data` | 19,184B | 19,168B | -16B |
| `.bss` | 21,232B | 21,232B | 0B |
| DGROUP静的ペイロード | 40,416B | 40,400B | -16B |
| ヒープ開始下限 | `0x9E00` | `0x9DF0` | -16B |

試験命令の解析・比較・文字列除去は実コード52B、アライン後コード48Bを削減した。
一時pending除去で`run_script_sjis()`のスタックは1,342Bから1,340Bへ2B減少した。
通常`#bg`化とインターレース予定側の除外は実コード24B増、失敗フォールバックは66B増である。

非ゼロfar textはない。全画面プレーンコピーのスタックは20B、完成シーンAPIは30B、
UIコールバックは22B、ロード復帰関数は8Bである。
